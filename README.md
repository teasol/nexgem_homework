# 📑 NEXGEM Algorithm Homework

NEXGEM 기초 알고리즘 과제 제출을 위한 저장소입니다. 리트코드(LeetCode) 문제 풀이를 진행하며, 문제 해결을 위한 사고의 흐름과 최적화 과정을 기록합니다.

## 📂 Folder Structure

과제는 주차별 폴더로 관리하며, 각 문제별로 하위 폴더를 생성하여 관리합니다.

```text
/
├── week1/                 # 1주차 과제 (예: Array, String)
│   ├── Problem_Name_1/    # 문제별 폴더
│   │   ├── debug.py       # 디버깅용 파일
│   │   ├── solution.py    # 최종 제출 코드
│   │   └── README.md      # 시도한 기록, 분석 및 리뷰
│   └── ...
├── week2/                 # 2주차 과제 (예: Stack, Queue)
└── README.md              # 메인 가이드 (본 파일)

```

## 📝 Submission Guidelines

각 문제 폴더의 README.md 에는 다음 내용을 포함합니다.

* LeetCode 문제 링크
* Trial History
* 각 버전의 approach 설명
* 최종 알고리즘의 시간 복잡도 설명
* 문제 분석 및 최종 리뷰

Trial History는 지우지 말고 이어서 작성합니다.

Approach는 각 버전마다 (commit 마다) 작성합니다.

매번 시간복잡도를 분석할 필요는 없으나, 최종 알고리즘의 시간 복잡도 설명은 필수입니다.

단순 문법 오류는 commit 및 기록 할 필요 없습니다. 기본적인 디버깅 후에 문법 오류가 없음을 확인하고 commit하는 습관을 들이도록 합시다.

단순히 정답을 맞히는 것이 목적이 아니라, **최적화 과정**을 보여주는 것이 중요합니다.

## 🤖 Generative AI Usage Guidelines

본 과제의 목적은 **'문제를 해결하는 사고 과정'**을 훈련하는 것입니다. AI를 도구로 활용하되, 아래 원칙을 준수해야 합니다.

### 1. 허용되는 범위 (Recommended)

* 개념 설명 및 원리 학습: 특정 알고리즘의 작동 원리를 이해하기 위해 설명을 요구하는 경우.
* 코드 디버깅 힌트: 작성한 코드에서 논리적 오류를 찾지 못할 때, 에러 메시지를 바탕으로 힌트를 얻는 경우.
* 주석 및 README 작성: 코드 설명 작성에는 AI를 적극 활용할 것.
* 시간 복잡도 검증: 본인이 계산한 복잡도가 맞는지 확인하는 경우.

### 2. 제한되는 범위 (Restricted)

* 정답 코드 복사 및 붙여넣기: 리트코드 문제 전체를 입력하고 AI가 생성한 코드를 그대로 제출하는 행위는 엄격히 금지합니다.
* 무비판적 수용: AI가 제시한 복잡도나 로직이 항상 정확하지는 않습니다. 반드시 본인이 검증하고 이해한 내용만 기록해야 합니다.

### 3. AI 활용 시 기록 방법 (Disclosure)

AI의 도움을 받아 시도(Trial)를 진행했다면, 해당 버전의 README.md 내 Approach에 반드시 명시하고, 어떤 점이 개선 되었는지 적어야 합니다.

작성 예시: 1시간 동안 생각했는데 도저히 시간 복잡도를 줄일 수 있는 방법이 생각이 안 나서 Gemini한테 물어봤음. 단순히 처음부터 끝까지 확인하는 게 아니라 two-pointer라는 방식이 있다는 것을 알게 되었음.

### 4. 핵심 원칙: "Think First, Ask Later"

최소 20분 이상은 외부 도움 없이 스스로 문제를 분석하고 설계를 시도합니다.

AI에게 질문할 때는 단순히 "정답을 알려줘"가 아니라, **"내 접근 방식에서 시간 복잡도를 더 줄일 수 있는 아이디어가 있을까?"**와 같이 구체적으로 질문합니다.

AI가 작성해 준 코드가 있다면, 반드시 한 줄씩 직접 타이핑하며 로직을 분석해야 합니다.
